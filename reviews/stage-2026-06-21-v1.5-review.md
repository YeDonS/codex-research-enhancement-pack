# 阶段 Review：2026-06-21 v1.5

## 本轮真实任务

使用 WiscKey 和 HotRAP 两篇全文及用户已有 Obsidian 笔记，评估“分层 value storage、热 value 提升、range hotness、2 GB 快层和 GC 协同”是否值得继续。

## 真正提升 Codex 科研能力的部分

- `literature-evidence-reader` 没有停在摘要层，而是把架构、range-query 弱点、GC 安全顺序、并发竞态、容量限制和 ablation 绑定到页码/图表。
- `research-question-council` 识别出“热 value promotion”已经被 HotRAP 大量覆盖，淘汰了最容易重复已有工作的表述。
- `experiment-design-planner` 把“降低写放大”从预设结论改成可失败假设，并拆出 promotion、fast/slow GC、mapping update 和 demotion 的物理写入。
- 用户提出的“GC 时刚好搬迁”被转化为可验证机制：GC 可摊销 source read，但不能保证 key-order layout，也不是零成本。
- 2 GB 不再被当作孤立常数，而是绑定 DB ratio、hot-set capacity 和 GC headroom。

## 只是泛泛建议或应删除的部分

- “冷热分层会提升性能”没有 workload、容量和成本边界，删除。
- “更新 key index 可降低写放大”方向可能相反，改成 direct pointer 与 range manifest 对照。
- “范围扫描就是热范围”会导致 sequential flooding，改成 repeated reuse + flood guard。
- “GC 顺便搬迁”忽略排序、publish 和 live-copy 成本，不能进入 skill 的默认建议。
- 仅凭 HotRAP future work 不能声称 novelty，必须补系统性 related-work 检索。

## 风险与限制

- 本轮 related-work 只是两篇种子论文加定向边界检查，不是 systematic review。
- 尚未选择原型 codebase，也没有运行存储 benchmark 或 crash test。
- 15/16 rubric 分数评价的是任务产物质量，不是系统方法性能。
- 公开验证文件只保留释义和来源定位，未提交私有 PDF 或 Obsidian 原文。
