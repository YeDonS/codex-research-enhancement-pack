# 阶段 Review：2026-06-21

## 真正提升 Codex 能力的方法

1. 把经验转为 skill 的元流程。
   - 有明确的工程化链路：调研、分析、计划、开发、验证、测试、审计验收、总结迭代。
   - 能直接提升长期复用能力，已转化为 `xhs-method-ingester`。

2. 独立上下文 review。
   - 解决主对话长上下文自证正确的问题。
   - 对科研中的证据检查、方法审查、实验计划 review 有明显价值。

3. 文献知识库从“摘要”改为“证据卡/方法卡/变量表/来源定位”。
   - 直接针对科研产物质量。
   - 能避免 Obsidian 里堆不可用摘要。

4. PR/handoff 作为跨工具协作通道。
   - 解决人工搬运文件和状态混乱。
   - 可迁移到代码复现、数据分析、论文写作和长期任务。

5. 自动化闭环：计划、学习/研究页、练习或实验记录、归档、日志。
   - 适合长期科研推进。
   - 必须保持低负担和可审查。

## 只是泛泛建议或证据不足的方法

- “科研工作流技能全流程公开”这条只有话题标签，没有可执行正文。本阶段只作为弱信号，不生成独立能力。
- “自动找 paper 的 skill”正文没有公开方法细节，只能提炼出低门槛、反馈入口和文献检索需求，不能生成完整 paper finder。
- “工具链越复杂越好”这类隐含倾向应删除，保留“是否减少人工负担”作为判断指标。

## 需要重写或收窄的方法

- LLM Council 原始提示偏通用决策，已重写为科研选题/假设/实验计划审查，不用于日常闲聊。
- Codex 实战 10 条范围太广，已拆分为长期任务管理、skill 封装、计划模式、自动化四类，不单独作为一份大而全 skill。
- GitHub 协作流程适合代码和可版本化文档；PPT、PDF、视频、大数据文件应走外部通道，只在 handoff 记录位置。

## 删除规则

后续遇到以下内容，不转化为 skill：

- 只有工具名，没有操作步骤。
- 只有情绪表达或效果宣称，没有输入、输出、验证。
- 鼓励全自动但没有失败边界。
- 要求 Codex 长时间跑大任务，却没有阶段检查点。
- 产物只能增加人工审核负担，不能形成可复用知识。
